#73 - Features Page Category Tabs
What & Why
Add category-based tabs to the top of the /features page so users can quickly view features filtered by department (Sales, Programming, Production) or see everything at once.

Done looks like
Four tabs appear at the top of the /features page: All, Sales, Programming, Production
The All tab displays the features page exactly as it works today — with the existing filter bar, grouping options, and current default behavior (grouped by category)
Each category tab (Sales, Programming, Production) filters features to only those belonging to that category and groups them by status. The existing filter bar remains available within each tab for additional filtering (e.g., by status)
Tab selection is managed via local state (no URL changes needed)
The tabs match categories by name from the app_feature_categories table. If a category name doesn't exist in the database, the tab still renders but shows an empty state
The category tabs should be hardcoded to these three specific names — they are not dynamically generated from all categories
Out of scope
Making the tab list configurable or admin-editable
Adding new categories to the database (they should already exist; if not, the tabs just show empty)
Changing any other page behavior beyond adding the tab filtering layer
Tasks
Add a tab bar (using the existing Tabs component) above the current filter bar with four tabs: All, Sales, Programming, Production.
On the All tab, preserve the current page behavior exactly — default grouping by category, full filter bar.
On category-specific tabs, filter the features list to only features whose category.name matches the tab label, and override the grouping to "status". The filter bar should still work for additional status filtering within that category view.
Look up category IDs from the fetched categories query by matching on name, so the filtering works against categoryId.
